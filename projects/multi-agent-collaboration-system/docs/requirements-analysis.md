**版本**: v0.1  
**日期**: 2026-02-08  
**状态**: 草稿

## 1. 核心用例

### 1.1 任务分环节处理（Pipeline Collaboration）

**场景描述**:  
复杂任务需要多个专业 Agent 协作完成，每个 Agent 负责一个环节。

**示例**:
- **内容创作流水线**:
  1. Agent A: 搜集素材（web_search, web_fetch）
  2. Agent B: 撰写初稿（writing specialist）
  3. Agent C: 审校优化（editor）
  4. Agent D: 格式化发布（formatter）

- **代码开发流水线**:
  1. Agent A: 需求分析（product manager）
  2. Agent B: 架构设计（architect）
  3. Agent C: 编码实现（coder）
  4. Agent D: 测试验证（tester）

**核心需求**:
- ✅ 顺序执行（上一个完成才启动下一个）
- ✅ 上下文传递（每个 Agent 能看到上一个的输出）
- ✅ 失败重试（某个环节失败时能重试或跳过）
- ✅ 进度可视化（用户能看到当前在哪个环节）

---

### 1.2 并行协作（Parallel Collaboration）

**场景描述**:  
多个 Agent 同时工作在不同子任务上，最后汇总结果。

**示例**:
- **市场调研**:
  - Agent A: 调研竞品 A
  - Agent B: 调研竞品 B
  - Agent C: 调研竞品 C
  - Agent D: 汇总分析（等待 A、B、C 完成）

- **多平台发布**:
  - Agent A: 发布到 Twitter
  - Agent B: 发布到 Telegram
  - Agent C: 发布到 Discord
  - Agent D: 收集反馈

**核心需求**:
- ✅ 并发执行（多个 Agent 同时运行）
- ✅ 结果聚合（收集所有结果后汇总）
- ✅ 超时处理（某个 Agent 超时不应阻塞整体）
- ✅ 部分失败容忍（3/4 成功也能继续）

---

### 1.3 游戏场景 - 狼人杀（Game Scenario）

**场景描述**:  
多个 Agent 扮演不同角色，遵守游戏规则，进行推理和对话。

**游戏流程**:
1. **初始化**: 分配角色（狼人、村民、预言家、女巫等）
2. **夜晚阶段**: 狼人投票杀人，预言家查验身份
3. **白天阶段**: 所有玩家发言、推理、投票
4. **循环**: 重复夜晚-白天，直到游戏结束

**Agent 能力要求**:
- 🎭 角色扮演（根据身份调整策略）
- 🧠 推理能力（分析发言，识别矛盾）
- 💬 对话能力（说服、欺骗、辩论）
- 📊 记忆管理（记住历史发言和投票）

**核心需求**:
- ✅ 多轮对话（支持复杂的轮流发言）
- ✅ 状态同步（所有 Agent 看到相同的游戏状态）
- ✅ 规则引擎（强制执行游戏规则）
- ✅ 观战模式（用户可旁观或参与）

## 2. 技术挑战

### 2.1 Agent 通信

**问题**: 如何让 Agent 之间高效通信？

**方案对比**:

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **共享文件** | 简单、可追溯 | 性能差、无实时性 | 离线任务、存档 |
| **Message Queue** | 解耦、异步 | 复杂度高 | 高并发、分布式 |
| **Session Send** | OpenClaw 原生、简单 | 仅本地 | 单机协作 |
| **WebSocket** | 实时、双向 | 需要服务端 | 游戏、聊天 |

**推荐**: 先用 **Session Send**（MVP），后续可扩展 WebSocket。

---

### 2.2 任务调度

**问题**: 谁来决定下一个该哪个 Agent 执行？

**方案对比**:

| 方案 | 描述 | 优点 | 缺点 |
|------|------|------|------|
| **中央调度器** | Master Agent 控制全局 | 简单、可控 | 单点故障 |
| **去中心化** | Agent 自主协商 | 灵活、健壮 | 复杂、难调试 |
| **混合模式** | 关键节点集中，细节分布 | 平衡 | 设计复杂 |

**推荐**: **中央调度器**（MVP），Master Agent 负责协调。

---

### 2.3 状态管理

**问题**: 如何让所有 Agent 看到一致的状态？

**挑战**:
- 并发更新冲突
- 状态回滚
- 历史追溯

**方案**:
- 使用 **事件溯源 (Event Sourcing)**:
  - 所有改变记录为事件（Event Log）
  - Agent 读取事件重建状态
  - 天然支持历史回放和调试

**示例**:
```json
{"event": "player_speak", "player": "Agent1", "content": "我是好人", "timestamp": 1234567890}
{"event": "player_vote", "player": "Agent2", "target": "Agent3", "timestamp": 1234567891}
```

---

### 2.4 上下文管理

**问题**: 如何避免 Agent 上下文爆炸？

**挑战**:
- 游戏可能 10+ 轮，每轮 5+ 玩家发言 → 50+ 条消息
- 传统方式会超出上下文窗口

**方案**:
1. **关键信息摘要**: 每轮结束生成摘要，下一轮只传摘要
2. **角色专属记忆**: 每个 Agent 只记录与自己相关的信息
3. **外部知识库**: 把历史存到文件/数据库，需要时检索

**推荐**: **摘要 + 检索** 混合模式。

## 3. MVP 范围确定

### 3.1 MVP 目标

**一句话**: 实现 **3 个 Agent 完成简单流水线任务**。

**具体目标**:
- Agent A: 搜索一个问题的答案
- Agent B: 把答案改写成推文
- Agent C: 审核推文并打分

**成功标准**:
- ✅ 3 个 Agent 顺序执行
- ✅ 上下文正确传递
- ✅ 金哥能看到每个 Agent 的输出
- ✅ 总耗时 < 2 分钟

---

### 3.2 不在 MVP 范围内

❌ 并行执行  
❌ 失败重试  
❌ 狼人杀游戏  
❌ Web UI  
❌ 分布式部署  

这些留到 v1.0 或 v2.0。

---

### 3.3 技术选型（MVP）

| 组件 | 技术 | 理由 |
|------|------|------|
| **Agent 框架** | OpenClaw `sessions_spawn` | 原生支持，无需额外依赖 |
| **通信方式** | `sessions_send` | 简单、同步、可靠 |
| **调度机制** | Master Agent 脚本 | 手写控制逻辑，灵活 |
| **状态存储** | JSON 文件 | 简单、易调试 |
| **语言** | Python | 金哥熟悉，生态丰富 |

---

### 3.4 MVP 实现计划

**Step 1**: 设计 Master Agent 脚本  
**Step 2**: 创建 3 个 Worker Agent 配置（search, rewrite, review）  
**Step 3**: 实现任务流程控制逻辑  
**Step 4**: 测试端到端运行  
**Step 5**: 优化输出展示  

**预计时间**: 2-3 小时
