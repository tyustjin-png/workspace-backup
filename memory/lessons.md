## 2026-03-05 监控频率过高导致噪音

**情况：** 设置了端口监控（每15分钟）和内存监控（每10分钟）的cron任务

**错误：** 没有考虑到频繁提醒会打扰金哥，违背了HEARTBEAT.md里"每小时检查"的优化原则

**后果：** 
- 18:15-19:00 期间触发了7次无意义的提醒
- 金哥感到烦躁
- token浪费

**教训：** 
- 创建定时任务前，先检查是否与现有机制重复
- 监控类任务应该"异常才通知"，不是"定期汇报正常"
- 频率设置要符合实际需求（系统监控1小时够了）

**规则：** 
> 定时任务原则：必要性 > 频率。能合并到heartbeat的就不要单独创建cron。

## 2026-03-09 Heartbeat窗口推送机制失效（L019）

**情况：** 10:00 cron生成了日程汇报文件，但到10:13才补发

**错误：** Heartbeat在10:00-10:10窗口期间没有触发（或触发了但没执行推送逻辑）

**后果：** 
- 金哥在10:13问"为啥没有当日报告"
- 昨天刚修复的问题又出现了
- 验证了架构设计有缺陷

**根本原因：**
- **Heartbeat触发频率不可控**（由OpenClaw系统控制，可能间隔>10分钟）
- **被动等待 ≠ 主动推送**（依赖heartbeat触发 vs cron直接推送）
- **时间窗口设计不合理**（10分钟窗口，但heartbeat可能不在窗口内触发）

**教训：** 
- **定时通知不应依赖heartbeat** - heartbeat是巡检机制，不是定时推送器
- **Cron应该直接推送** - 生成文件后立即调用message工具发送
- **Heartbeat只负责兜底** - 检查是否有遗漏的通知，而非主力推送

**解决方案：**
1. ✅ 立即实施：修改 crontab，让 cron 直接调用 `openclaw message` 推送
2. ✅ Heartbeat 降级为兜底检查（发现遗漏才补发）
3. ✅ 删除时间窗口逻辑（不再依赖被动触发）

**规则：** 
> 定时推送 = Cron主动发送 + Heartbeat兜底检查。不要用被动机制做主动任务。

**优先级：** 🔴 高 - 立即修复，否则每天重复遗漏

## 2026-03-09 延迟修复导致用户体验差（L021）

**根本原因：**
- **过度优化token消耗** - 为了省token只设置1次审计

- **缺少用户视角** - 只想着技术实现，没站在用户角度思考

**教训：** 
- **快速反馈 > 完美设计** - 30分钟修复比13小时后修复重要得多
- **用户体验 > 技术优雅** - 多几个cron jobs无所谓，体验好才是关键
- **关键任务需要实时验证** - 不能假设一切正常

**解决方案（已实施）：**
1. ✅ 新增关键时间点验证审计（08:30/10:30/15:30/19:30）
2. ✅ 任务执行后30分钟验证，发现问题立即修复
3. ✅ 21:00完整审计降级为兜底检查+每日反思
4. ✅ 平均延迟从13小时降到30分钟

**新机制：**
```
任务执行 → 30分钟后验证 → 发现问题立即修复

08:00 定投执行 → 08:30 验证
10:00 日程推送 → 10:30 验证
15:00 日程推送 → 15:30 验证
19:00 日程推送 → 19:30 验证
21:00 完整审计（兜底）
```

**规则：** 
> 关键任务必须在执行后短时间内验证。延迟修复 = 体验灾难。

**优先级：** 🔴 高 - 直接影响用户体验和信任度

**触发者：** 金哥的质疑（感谢！）

## L022 - 不要信任中间状态，要端到端验证
**日期：** 2026-03-10
**场景：** 日程汇报cron显示delivered=true但Telegram没收到，验证审计Agent被骗
**问题：** 
1. cron的announce模式delivered状态不可靠
2. 验证审计Agent信任了这个状态，没有独立验证
3. 信任链断裂：cron说成功 → 审计信任 → 实际失败

**教训：** 
- 验证必须是端到端的，不能只检查中间状态
- "宁可重复也不能漏发" —— 验证任务应该直接执行，而非检查状态
- 任何"delivered=true"都要质疑，只有message工具返回ok才是真正送达

**修复：** 验证审计改为直接用message工具发送，而不是检查cron状态

## L023 - 任务完成消息必须写入文件
**日期：** 2026-03-14
**场景：** 金哥09:21报告运动完成，10:00日程推送仍显示运动未完成
**问题：** 
1. 收到金哥的任务完成消息后，只口头回复"扎实"
2. 没有更新日程文件 `memory/daily-schedule/YYYY-MM-DD.md`
3. 定时推送读取旧文件，显示错误状态

**根因：**
- 把任务完成当"聊天"处理，不是当"状态更新"处理
- 缺少标准化流程：收到完成消息 → 更新文件 → 回复确认

**教训：** 
- **先写后说** —— 收到任务完成消息时，先更新文件，再回复
- 状态变更必须持久化，不能只存在对话里
- 定时任务依赖文件状态，对话内容不会被读取

**修复规则 R024：**
```
收到任务完成消息时：
1. 立即更新 memory/daily-schedule/YYYY-MM-DD.md
2. 标记 [x] + 完成时间
3. 然后再回复确认
```

**效用评分：** 10/10（核心流程问题）
