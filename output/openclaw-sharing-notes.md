## 一、模型按任务分级，省钱又提效

### 核心观点

不是所有任务都需要最强模型。把模型当员工分级——能让实习生干的活，别请总监。

### 三层模型分配

| 层级 | 模型 | 价格 | 适用场景 |
|------|------|------|----------|
| 执行层 | Haiku | $0.25/M | 固定逻辑、跑脚本、格式化输出 |
| 处理层 | Sonnet | $3/M | 信息处理、日报生成、搜索总结 |
| 思考层 | Opus | $15/M | 深度分析、策略制定、创作任务 |

### 实际案例

我们系统跑着18个定时任务。仔细一看，6个是纯固定逻辑——日程文件创建、定投执行、日记提醒等。逻辑完全一样，不需要任何"智能"。

原来全用Sonnet，改完后6个切Haiku，**日常成本降了60%以上**。效果零差异。

### 踩坑经验

- **别一刀切**。有人把所有任务都切最便宜的，日报质量断崖下跌。关键是分清哪些需要智能，哪些不需要。
- **Haiku不是万能的**。涉及"理解上下文""做判断"的任务会翻车。它只适合确定性逻辑。
- 省下的钱给真正需要思考的任务上Opus，整体效果反而更好。

## 二、能用脚本就别用Agent

### 核心观点

Agent不是银弹。能用脚本解决的问题，套个Agent壳子只会更慢、更贵、更不可控。

### 三层金字塔

```
        ┌─────────┐
        │  Agent  │  多步推理、动态策略
       ┌┴─────────┴┐
       │   Skill   │  LLM单步处理
      ┌┴───────────┴┐
      │    脚本     │  确定逻辑、固定流程
      └─────────────┘
```

### 判断标准——问自己三个问题

1. **输入输出确定吗？** 确定 → 脚本
2. **需要LLM理解，但一步完成？** → Skill
3. **需要多步推理、动态调整？** → Agent

### 我们的实际对照

**脚本搞定的：**
- 定投执行：读指数→算金额→调API下单。逻辑完全确定。
- 日程文件创建：复制模板、改日期。纯字符串操作。
- Web3雷达：脚本抓数据，只在"打分"环节用LLM。

**必须Agent的：**
- 1244篇笔记深度分析：逐篇阅读、提炼观点、发现跨篇章联系。
- 投资研判：搜索实时信息+多维推理+综合判断，每次不同。

### 踩坑经验

- 最常见的错误：把简单任务包装成Agent。定投如果让Agent来做，它可能"思考"一下决定今天不买了——这不是你想要的。
- **脚本的优势是确定性**。同样输入永远同样输出，不会因为模型升级突然改行为。
- **混合架构最实用**。数据采集用脚本保证稳定，只在需要"判断"的环节调LLM。

## 三、并行任务分发是最大的效率杠杆

### 核心观点

一个AI读1244篇文章要好几天。13个AI同时读，几小时搞定。并行分发是ROI最高的能力。

### 实际案例

我有1244篇个人笔记，7年积累，需要AI逐篇阅读分析。

**第一轮：Sonnet快速扫描**
- 1244篇分3批，3个Sonnet并行读
- 产出3份分析报告+1份总分析
- 几小时搞定

**第二轮：Opus深度阅读**
- 1244篇分13批，每批约100篇
- 每批一个独立Opus session并行跑
- 每批产出约20000字深度分析
- 13个AI同时干活

核心流程：**一条指令 → 拆分多个AI并行 → 结果自动回报**

### 踩坑经验

- **批次大小要合理**。太大context会爆，太小开销不值。实测100篇/批是甜点。
- **先用便宜模型跑一轮**。第一轮Sonnet快扫，发现问题先修，再上Opus精读。别一上来就烧最贵的钱。
- **统一输出格式**。否则13个session产出的报告格式不一，合并是噩梦。
- **注意API并发限制**。13个Opus同时跑可能触发限流，必要时分两波。

## 四、记忆系统三层架构

### 核心观点

大多数人用AI只做L1（记录），从不蒸馏。真正的价值在于把经验转化为可复用的判断原则——本质上是教会AI你的决策风格。

### 三层设计

```
L3 决策公理 (Axioms)     ← 最高抽象，指导决策
L2 模式提炼 (Reflector)  ← 从重复中识别规律
L1 日常观察 (Observer)   ← 每天事件流水账
```

**L1 日常观察**
每天发生了什么，做了什么决定，遇到什么问题。原始记录，不加工。

**L2 模式提炼**
从重复事件中识别规律。比如：助手改文件前不先读文件格式，连续翻车两次 → 提炼出硬性规则"改文件前必须先读文件"。

**L3 决策公理**
我们从1244篇笔记中蒸馏出**46条决策公理**，7个类别：投资决策、风险管理、信息处理、人际关系、个人成长、内容创作、团队管理。每条有证据支持和置信度标注。

### 关键洞察

- **L1到L3是蒸馏过程**。1244篇日记 → 提炼出模式 → 凝结为46条公理。信息密度提升几百倍。
- **本质是教会AI你的决策风格**。面对新决策时，AI不是从零推理，而是基于你的公理体系来判断。
- **大多数人卡在L1**。记了一堆日记从来不回头看。AI可以帮你蒸馏——这可能是AI对个人知识管理最大的价值。

### 踩坑经验

- AI提炼的公理**必须人工审核**。它会过度泛化，也会编造不存在的规律。
- **置信度标注很重要**。让AI知道哪些原则硬执行，哪些只是参考。
- **定期更新**。人会变，三年前的决策风格可能已经进化了。

## 五、公众号文章直接API抓取

### 核心观点

微信公众号是中文互联网最大的内容孤岛。有个开源工具能直接API抓取全文，配合AI做内容分析非常好用。

### 工具

- **在线版**：https://down.mptext.top
- **开源项目**：wechat-article/wechat-article-exporter
- **核心能力**：免登录，一个API调用返回文章完整正文

传入微信文章链接，返回标题、作者、全文内容、发布时间。直接喂给AI处理。

### 实际案例

今天用这个API抓取了《能用脚本就别用Agent》全文，直接让AI做内容分析和对照。从抓取到分析完成不到5分钟。

### 扩展用途

- **批量抓取**：某个公众号全部历史文章，做知识库
- **内容分析**：抓取后让AI做摘要、提炼、翻译
- **竞品监控**：定期抓取关注的公众号，AI筛选有价值内容

### 踩坑经验

- 不是所有文章都能抓，付费阅读/特殊权限的可能失败
- 原始HTML包含大量样式，喂AI前最好先转纯文本，减少token浪费
- 别短时间大量请求，尊重开源作者的服务器
