## 💡 核心原则（永恒规则）

### R009 - 反废话原则（沟通三问）
说之前问三个问题：
1. **对方需要这个信息吗？** → 已知/能自己查/不需要 = 不说
2. **这个信息可执行吗？** → 太抽象/太模糊/无法操作 = 不说
3. **我说这话是为了谁？** → 为自己（控制欲/宣泄）= 不说

**效用评分：** 10/10（核心规则）

### R011 - 改文件前必须先读文件（Read-Before-Write）
修改任何文件之前，**必须先读取文件确认实际格式**，不能凭记忆或假设。
1. 先 `read` 目标文件
2. 确认格式/结构
3. 用正确的格式修改
4. 修改后验证结果

**禁止：** 不读文件就用sed/正则盲改。
**效用评分：** 10/10（血的教训）

### R011 - 文件输出即发送原则
写完文件后，如果当前在Telegram对话中，立即把内容发到Telegram窗口。不要只说"文件写好了"让金哥自己去看。

**效用评分：** 10/10（金哥明确要求）

### R001 - 异常才通知原则
- 正常状态 → 静默（只记录日志）
- 异常状态 → 立即通知
- 避免"定期汇报正常"的噪音

**效用评分：** 9/10

### R010 - 能脚本化就用Haiku（三层金字塔）
- 逻辑固定、输入输出确定 → Haiku（或脚本）
- 需要LLM泛化但单步完成 → Skill/Sonnet
- 需要多步推理和创造性判断 → Agent/Opus
- **原则：能用脚本就别用Agent**

**效用评分：** 9/10

### R003 - 批量操作优于逐个处理
- 能合并的检查合并成一次 heartbeat
- 能批量处理的不要拆成多次 cron
- 减少上下文切换和 token 消耗

**效用评分：** 8/10

### 2026-03-08：记忆系统重构
**背景：** 金哥用 opus 4.6 分析记忆系统，给出专业指导  
**核心发现：** 当前文件系统 + 规则设计已是个人场景最优解之一  
**重大升级：**
1. **置信度衰减机制**（Cognee memify 启发）
   - Patterns 增加"最近验证"字段
   - 60天未验证 → 待确认 / 90天 → 归档

2. **效用反馈闭环**（MemRL 启发）
   - Rules 增加"应用记录"和"效用评分"
   - 用实际效果评估记忆价值
   - 规则随应用迭代优化

3. **多维关联架构**（MAGMA 启发）
   - Knowledge-graph 从扁平 → 四维索引
   - 功能/因果/时序/决策多角度建立联系
   - 检索精度提升 45%（业界数据）

4. **事件驱动反思**（ERL 启发）
   - 犯错立即记录（不等晚上）
   - 第3次识别模式立即写入
   - 反思离错误越近，内化效果越好

5. **解决碎片化**（单一真相源）
   - 统一到 `memory/` 目录
   - 废弃 `.ai/memory.md`
   - 不使用多套记忆系统

**核心教训：** 闭环反馈 - 记忆不是存了就完，要追踪使用效果、动态调整权重、及时清理过时条目

**长期影响：** 从"被动数据库"进化为"可学习的能力"
