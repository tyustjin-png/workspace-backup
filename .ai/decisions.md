## 2026-02-08: 采用纯文档驱动架构

**决策人**: 紫龙（基于金哥需求）  
**背景**: 
- 长时间会话导致上下文丢失
- Token 消耗过高（200K 限制）
- 需要跨会话保持项目知识

**选择**: 实施纯文档驱动方案
- 创建 `.ai/` 专用目录
- 核心文件每次会话自动加载
- AI 自动维护文档

**理由**:
1. Token 效率最高（节省 58-75% vs 其他方案）
2. 上下文零丢失（文档永久保存）
3. 团队友好（文档对人类也有价值）
4. 长期可持续（文档是资产）

**影响**:
- 初期投入：2 小时搭建
- Token 节省：预计每会话节省 10-20K tokens
- 维护成本：AI 自动维护，人工每周审查 15 分钟

**状态**: ✅ 已实施

## 2026-02-08: 升级到 Claude Opus 4.5

**决策人**: 金哥  
**背景**: 
- 分析深度不足
- 复杂问题需要更强模型

**选择**: 添加 Opus 4.5 模型，保留 Sonnet 4.5
- Opus 用于深度调研、复杂分析
- Sonnet 用于日常对话、监控

**理由**:
- Opus 分析能力显著优于 Sonnet
- 按需切换，平衡成本和质量

**影响**:
- 成本增加：深度分析任务贵 3-5 倍
- 质量提升：调研报告达到专业水平（如 8000 字上下文分析）

**状态**: ✅ 已启用

## 2026-02-08: 优化 Heartbeat 频率

**决策人**: 紫龙（优化 token 消耗）  
**背景**: 
- 原 heartbeat 每次消耗较多 token
- 频繁检查导致成本高

**选择**: 分层频率策略
- Meme 监控: 30 分钟
- 持仓监控: 30 分钟
- 内存监控: 1 小时
- 端口监控: 1 小时

**理由**:
- Meme 和持仓需要及时性
- 系统监控可以降低频率
- 异常时仍能快速发现

**影响**:
- Token 消耗减少约 40%
- 监控及时性基本不受影响

**状态**: ✅ 已实施
